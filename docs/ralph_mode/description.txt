Claude Ralph Loop

프론트엔드에서의 Ralph Loop 선정 이유:
- 전체를 구현하기에는 직접 해보고 싶기도 하고, 클로드 코드 요금제도 부족
- 바이브 코딩의 결과물을 시각적으로 확인하고 검증할 수 있다는 장점

다만, 여러가지 현실적인 이유로 완전한 Ralph Loop랑은 조금 차이가 있었다. 
(1) 사용한 Prompt 및 구현 기획을 클로드 코드가 자동적으로, 클로드 세션과 대화를 하며 만들었고,
(2) Prompt가 상당히 구체적이었기 때문에 클로드 코드가 기능 측면에서는 구현해야할 plan이 이미 존재해야하는 채로, 세부적인 디자인과 코드 구현만을 맡았다.
(3) 또한 3번의 루프만을 진행하였다.

현실적인 이유: (1) backend 구조가 확정된 상태였기 때문에 이 부분에 대한 구체적인 전달이 필요, 
(2) 원하는 final output의 전체적인 그림이 어느정도 구체적으로 존재했으며, 
(3) 이를 수십 번의 loop로 알아서 구현하기를 기다릴 수 있는 시간 및 클로드 코드 비용(Pro 요금제)에 제한이 있었다.

그럼에도 불구하고 ai-usage-log.md에서 확인할 수 있듯이 루프를 거치며 발전하는 모습을 보였다. 

[결과물]
전반적으로 굉장히 잘 개발을 해주었다. 어느정도 원하는 디자인과 자연스러운 UI를 구현해준 모습을 확인할 수 있었다.

의외로 좋았던 점:
- 디자인이 내가 구상한 것과 거의 99% 일치했다.
- 간단한 버전의 FastAPI이기는 했지만, 그럼에도 FastAPI 엔드포인트도 잘 연결해주었다. 
- git commit을 따로 하드코딩하지 않고 프롬프트에 넣어두기만 했는데에도 매 루프를 커밋했다. 
- 여러 번의 루프를 거치며 발전하는 모습을 보였다.
- 프론트엔드 전체를 루프로 구축한 것 치고는 생각보다는 토큰을 많이 쓰지 않았다. (Pro 요금제 기준, 5시간 세션 25%)
