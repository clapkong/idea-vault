Claude Ralph Loop

프론트엔드에서의 Ralph Loop 선정 이유:
- 전체를 구현하기에는 직접 해보고 싶기도 하고, 클로드 코드 요금제도 부족
- 바이브 코딩의 결과물을 시각적으로 확인하고 검증할 수 있다는 장점

다만, 여러가지 현실적인 이유로 완전한 Ralph Loop랑은 조금 차이가 있었다. 
(1) 사용한 Prompt 및 구현 기획을 클로드 코드가 자동적으로, 클로드 세션과 대화를 하며 만들었고,
(2) Prompt가 상당히 구체적이었기 때문에 클로드 코드가 기능 측면에서는 구현해야할 plan이 이미 존재해야하는 채로, 세부적인 디자인과 코드 구현만을 맡았다.
(3) 또한 3번의 루프만을 진행하였다.

현실적인 이유: (1) backend 구조가 확정된 상태였기 때문에 이 부분에 대한 구체적인 전달이 필요, 
(2) 원하는 final output의 전체적인 그림이 어느정도 구체적으로 존재했으며, 
(3) 이를 수십 번의 loop로 알아서 구현하기를 기다릴 수 있는 시간 및 클로드 코드 비용(Pro 요금제)에 제한이 있었다.

그럼에도 불구하고 ai-usage-log.md에서 확인할 수 있듯이 루프를 거치며 발전하는 모습을 보였다.

의외로 좋았던 점:
- 디자인이 내가 구상한 것과 거의 99% 일치했다.
- 간단한 버전의 FastAPI이기는 했지만, 그럼에도 FastAPI 엔드포인트도 잘 연결해주었다. 
- git commit을 따로 하드코딩하지 않고 프롬프트에 넣어두기만 했는데에도 매 루프를 커밋했다. 
- 여러 번의 루프를 거치며 발전하는 모습을 보였다.
- 프론트엔드 전체를 루프로 구축한 것 치고는 생각보다는 토큰을 많이 쓰지 않았다. (Pro 요금제 기준, 5시간 세션 25%)

  1. 진단 레이어가 계속 틀렸다                                                                                                                  
                                                                                                                                                
  증상이 "TOC가 같이 스크롤됨"으로 보였기 때문에, 매 시도마다 컴포넌트 레벨에서 고치려 했습니다:                                                
                                                                                                                                                
  ┌────────┬─────────────────────┬──────────────────────────────────┬─────────────┐                                                             
  │  시도  │      수정 위치      │               방법               │    결과     │                                                             
  ├────────┼─────────────────────┼──────────────────────────────────┼─────────────┤                                                             
  │ Loop 4 │ .result-toc         │ position: sticky                 │ 실패        │
  ├────────┼─────────────────────┼──────────────────────────────────┼─────────────┤
  │ Loop 5 │ 각 페이지 컴포넌트  │ min-height: 0 + overflow: hidden │ 부분 개선   │
  ├────────┼─────────────────────┼──────────────────────────────────┼─────────────┤                                                             
  │ Loop 6 │ .result-toc         │ overflow: hidden                 │ 오히려 악화 │
  ├────────┼─────────────────────┼──────────────────────────────────┼─────────────┤                                                             
  │ Loop 7 │ html / body / #root │ height: 100% + overflow: hidden  │ 해결        │
  └────────┴─────────────────────┴──────────────────────────────────┴─────────────┘                                                             
  
  실제 문제는 브라우저가 body 레벨에서 스크롤바를 그리고 있었던 것인데, 하위 컴포넌트에서 아무리 overflow: hidden을 걸어도 body의 네이티브      
  스크롤은 차단할 수 없습니다.                              
                                                                                                                                                
  ---                                                       
  2. min-height: 100vh vs height: 100%의 함정
                                                                                                                                                
  /* 기존 — 문제 있음 */
  body { min-height: 100vh; }   /* 최소 높이만 지정 → 내용이 넘치면 무한정 늘어남 */                                                            
  #root { min-height: 100vh; }  /* 동일 */                                                                                                      
                                                                                                                                                
  /* 수정 — 해결 */                                                                                                                             
  body { height: 100%; overflow: hidden; }   /* 높이 고정, 브라우저 스크롤 차단 */
  #root { height: 100%; overflow: hidden; }  /* 동일 */                                                                                         
                                                            
  min-height는 "최소 이 크기 이상"이라는 의미라 콘텐츠가 뷰포트보다 커지면 body가 늘어납니다. height: 100%는 뷰포트에 딱 고정됩니다. Flex       
  자식들의 overflow: hidden은 부모 체인이 height로 고정되어 있을 때만 유효합니다.
                                                                                                                                                
  ---                                                       
  3. 브라우저 테스트 없이 추론만으로 판단
                                                                                                                                                
  레이아웃 버그는 코드만 보고 "이렇게 하면 될 것 같다"고 판단하기 쉬운데, 실제로 브라우저가 어떤 레이어에서 스크롤바를 만들고 있는지는 직접 
  렌더링해봐야 알 수 있습니다. 각 수정이 로컬에서는 논리적으로 맞아 보였지만, 정작 브라우저 DevTools의 scroll 컨테이너 표시나 실제 화면을 확인할
   수 없어서 근본 레이어를 놓쳤습니다.

  요약하면: CSS 스크롤 격리는 반드시 html → body → #root → main 전체 체인이 height로 고정되어야 동작하는데, 가장 바깥쪽(body/root)을 건드리지
  않고 안쪽만 수정하는 패턴을 반복한 게 원인입니다.
