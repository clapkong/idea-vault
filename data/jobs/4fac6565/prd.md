# 캠퍼스 밥친구 매칭 서비스 "밥약"

## 1. 한 줄 정의
같은 학교 학생들이 식사 시간대와 메뉴 선호도를 기반으로 밥 먹을 친구를 매칭해주는 웹/모바일 서비스

## 2. 문제 정의
대학생들은 혼자 밥 먹기 싫지만 매번 같은 친구들과만 먹게 되고, 새로운 사람을 만나고 싶어도 어색함 때문에 먼저 제안하기 어렵습니다. 특히 복학생, 편입생, 1학년은 점심시간에 함께 밥 먹을 사람을 찾기 힘듭니다. 이 서비스는 "밥 먹자"는 가벼운 목적으로 자연스럽게 새로운 사람을 만날 기회를 제공하여, 캠퍼스 생활의 사회적 고립감을 해소하고 학내 커뮤니티를 활성화합니다.

## 3. 타겟 사용자
- 1차: 같은 대학 재학생 (학교 이메일 인증)
- 주요 페르소나:
  - 혼밥이 싫은데 매번 연락할 친구가 없는 학생
  - 새로운 친구를 사귀고 싶지만 먼저 말 걸기 어려운 학생
  - 복학/편입/1학년으로 아는 사람이 적은 학생
  - 같은 전공/관심사를 가진 사람과 네트워킹하고 싶은 학생

## 4. 핵심 기능

### 4.1 학교 이메일 인증 및 프로필 설정
학생증 이메일로 본인 인증 후, 학과/학년/성별/음식 선호도(한식/양식/중식/일식/분식)/식사량/채식 여부 등 간단한 프로필 작성

### 4.2 밥약 등록 및 매칭
"오늘 12시 학생식당 한식 먹을 사람" 형태로 일정 등록하면, 같은 시간대에 비슷한 선호도의 사용자와 자동 매칭 제안 (최대 4인)

### 4.3 매칭 확정 및 채팅
매칭 제안을 수락하면 그룹 채팅방 생성, 만남 장소 확정 및 간단한 자기소개 교환

### 4.4 후기 및 평점 시스템
식사 후 상대방에게 별점(1-5)과 한 줄 후기 작성, 노쇼 신고 기능으로 신뢰도 관리

### 4.5 관심사 태그 매칭 (선택)
"게임", "운동", "영화", "코딩" 등 관심사 태그 설정 시 같은 태그 사용자 우선 매칭

## 5. 기술 스택

### 프론트엔드
- **웹**: React.js + TypeScript, Tailwind CSS (반응형 디자인)
- **iOS 앱**: SwiftUI, Alamofire (네트워킹)

### 백엔드
- **서버**: Node.js + Express.js
- **데이터베이스**: PostgreSQL (사용자/매칭/평점 데이터 관리)
- **인증**: JWT 토큰 기반 인증, Nodemailer (이메일 인증)

### 공통
- **실시간 통신**: Socket.io (채팅)
- **배포**: AWS EC2 (백엔드), AWS RDS (DB), Vercel (웹 프론트), TestFlight (iOS 베타 배포)
- **협업**: Git/GitHub, Notion

## 6. 예상 일정

### 1-2주차: 기획 및 설계
- 요구사항 정의서 작성
- DB 스키마 설계 (사용자, 매칭, 채팅, 평점 테이블)
- API 명세서 작성
- Figma 와이어프레임 (기본 레이아웃만, 최소 디자인)

### 3-5주차: MVP 개발 (핵심 기능 1-3)
- 백엔드: 회원가입/로그인 API, 밥약 등록/조회 API
- 프론트: 회원가입/로그인 화면, 밥약 등록 화면
- iOS: 기본 뷰 구조 및 네트워킹 연동

### 6-8주차: 매칭 로직 및 채팅 구현
- 백엔드: 매칭 알고리즘 (시간대/선호도 기반), Socket.io 채팅 서버
- 프론트: 매칭 목록 화면, 채팅 화면 (웹/iOS)
- DB: 매칭 히스토리 저장

### 9-11주차: 평점 시스템 및 부가 기능
- 백엔드: 평점/후기 API, 노쇼 신고 로직
- 프론트: 프로필 화면 (평점 표시), 후기 작성 화면
- 관심사 태그 기능 추가 (시간 여유 시)

### 12-13주차: 테스트 및 배포
- 통합 테스트 (웹/앱/백엔드)
- 버그 수정 및 성능 최적화
- AWS 배포, 학교 내 베타 테스트 모집 (20-30명)

### 14주차: 최종 발표 준비
- 사용자 피드백 반영
- 발표 자료 작성 (문제 정의, 해결 방안, 기술 스택, 데모)

## 7. 주요 리스크

### 리스크 1: 초기 사용자 확보 어려움
- **대응**: 우선 자기 학과/동아리부터 시작, 오픈 채팅방에 홍보, 교수님께 학생 대상 공지 요청

### 리스크 2: 노쇼 및 악의적 사용자
- **대응**: 신고 기능 + 누적 신고 시 계정 정지, 평점 시스템으로 신뢰도 가시화, 초기엔 수동 모니터링

### 리스크 3: 실시간 채팅 구현 복잡도
- **대응**: Socket.io 공식 문서 및 튜토리얼 활용, 초기엔 단순 텍스트 채팅만 구현 (이미지/알림은 2순위)

### 리스크 4: 3개월 일정이 타이트함
- **대응**: MVP 우선순위 명확화 (기능 1-4는 필수, 5는 선택), 매주 스프린트 회의로 진행 상황 점검, 기능 축소 가능성 열어두기

### 리스크 5: 디자인 퀄리티 부족
- **대응**: Material-UI(React), SF Symbols(iOS) 등 기본 UI 라이브러리 활용, Tailwind CSS 유틸리티로 최소한의 깔끔한 디자인 유지, 무료 컬러 팔레트(Coolors) 사용
